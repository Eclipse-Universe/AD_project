## Exp 1 — contamination 0.32 (`'auto'` → 0.32)

**날짜**: 2026-06-30

**재현 커밋**: `d2327fe` — `cd src && python run_experiment.py` (EXP_NAME="exp1", CONTAMINATION=0.32)

**변경 사항**: Exp 0과 모든 게 동일, `IsolationForest`의 `contamination`만 `'auto'` → `0.32`로
변경. 가설: train 기준 threshold를 풀어주면 test에서 더 많이 "이상"으로 잡아 recall이 오를 것이다
(Exp 0 분석의 직접적인 후속).

**로컬 진단 (제출 전)**: predicted positive rate가 14.81% → **44.36%**로 상승 — 목표였던 32.2%를
12%p 초과. `contamination`이 train(100% 정상) 점수 분포 기준 threshold이기 때문에, 그 threshold를
test에 적용했을 때 정확히 같은 비율이 나온다는 보장이 없다는 점(Q&A 기록 참고)이 그대로 드러남.
test 점수 분포가 train보다 더 넓게 퍼져 있다는 뜻이며, 두 가지 가능성을 구분해야 함:
1. test의 실제 이상 run들이 점수를 크게 끌어내림 (recall이 크게 오르는 좋은 신호)
2. test의 정상 run도 train 정상과 미세하게 달라 threshold에 과민 반응 (precision이 크게 깎이는
   나쁜 신호)

**점수 (public)**: F1 0.5699, Accuracy 0.6704

**Confusion matrix 역산** (Exp 0와 동일한 방법, PP=315,112):

| | Exp 0 (`auto`) | Exp 1 (`0.32`) |
|---|---|---|
| TP | 93,573 | 155,128 |
| FP | 11,631 | 159,984 |
| FN | 134,996 | 74,164 |
| TN | 470,200 | 321,124 |
| Recall | 0.4094 | 0.6766 |
| Precision | 0.8894 | 0.4923 |
| F1 | 0.5607 | 0.5699 |
| Accuracy | 0.7936 | 0.6704 |
| 추정 실제 anomaly 비율 | 0.3217 | 0.3228 (두 실험이 독립적으로 거의 같은 추정치를 줌 — 신뢰도 ↑) |

**분석**:
1. **threshold 위치가 핵심 병목이 아니었다.** FP가 11,631 → 159,984로 13배 넘게 폭증하고 recall이
   41%→68%로 크게 올랐는데도 F1은 +0.0092밖에 안 올랐다 — recall/precision이 거의 1:1로 상쇄되며
   F1 곡선이 이 구간에서 평평하다는 뜻.
2. **모델의 anomaly score 자체의 분리력이 약하다는 신호.** predicted positive rate(44.36%)가 이미
   추정 실제 비율(32.3%)을 넘어섰는데도 recall이 67.66%에 그쳤다 — score가 정상/이상을 잘
   구분했다면 실제 비율 근처로 threshold를 맞췄을 때 recall이 100%에 가까워야 한다. 그렇지 않다는
   건 새로 잡힌 FP 다수가 진짜 이상이 아니라 그냥 threshold 근처의 애매한 정상 데이터라는 뜻 —
   Exp 0에서 보류했던 가설 2번(row 독립 가정의 한계, 시간 정보 미사용)이 이 결과로 뒷받침됨.
3. 결론: Tier 1(threshold 튜닝)만으로는 한계에 가까워 보인다. 한 지점(0.32)만으로 단정하기엔
   이르지만, recall/precision이 거의 대칭적으로 상쇄된 점이 강한 신호.

## Exp 2 — contamination 0.23 (Exp 1 결과로부터 외삽)

**날짜**: 2026-06-30

**재현 커밋**: `82837ad` — `cd src && python run_experiment.py` (EXP_NAME="exp2", CONTAMINATION=0.23)

**변경 사항**: Exp 1의 predicted positive rate(44.36%)가 목표(32.2%)를 1.39배 초과했으므로,
`0.32 / 1.39 ≈ 0.23`으로 거칠게 외삽해 다음 값을 잡았다. Tier 1(threshold 튜닝)이 한계에
부딪혔는지, 아니면 아직 더 나은 지점이 남아있는지(F1 곡선의 모양)를 확인하기 위한 세 번째 점.

**로컬 진단 (제출 전)**: predicted positive rate 36.13% — 목표(32.2%)와의 오차가 Exp 1(12%p)보다
줄어든 4%p. 외삽 방향은 맞았지만 여전히 정확히 일치하지는 않아 contamination↔test rate 관계가
완전한 선형은 아님을 다시 확인.

**점수 (public)**: F1 0.5812, Accuracy 0.7133 — **지금까지 최고 F1**

**Confusion matrix 역산** (PP=256,696): TP=141,325, FP=115,371, FN=88,301, TN=365,403 →
Recall 0.6155, Precision 0.5506, 추정 실제 anomaly 비율 0.3232 (Exp 0/1과 거의 동일, 재확인됨)

**Exp 0/1/2 종합 비교**:

| | predicted rate | F1 | recall | precision |
|---|---|---|---|---|
| Exp 0 (`auto`) | 14.81% | 0.5607 | 0.4094 | 0.8894 |
| Exp 1 (`0.32`) | 44.36% | 0.5699 | 0.6766 | 0.4923 |
| Exp 2 (`0.23`) | 36.13% | **0.5812** | 0.6155 | 0.5506 |

**분석**:
1. **F1이 역U자(봉우리) 모양** — predicted rate가 추정 실제 비율(32.3%)에 가장 가까운 Exp 2가 F1
   최고점. "predicted rate ≈ true rate" 휴리스틱이 실제로 유효했음이 확인됨.
2. 그러나 봉우리의 **높이 자체가 0.58 부근에 머물러 있다.** 거의 정확한 threshold(Exp 2)에서도
   recall 61.55%/precision 55.06%로 둘 다 평범 — threshold를 더 정밀하게 맞춰도 추가로 얻을 수
   있는 F1은 크지 않을 것으로 추정됨 (이미 봉우리 근처).
3. Exp 1에서 제기했던 가설(row 단위 anomaly score의 분리력 자체가 약함)이 다시 뒷받침됨 — Tier 1
   (threshold 튜닝)은 거의 소진된 것으로 판단, Tier 2(시간 정보 feature)로 넘어갈 근거가 충분함.

## Exp 3 — diff feature 추가 (원본 52개 → 원본 + 변화량 = 104개)

**날짜**: 2026-07-01

**재현 커밋**: Exp 3 코드(run_experiment.py)는 결과 확인 후 Exp 4로 바로 수정돼 별도 커밋 없음.
`preprocess.py`의 `add_diff_features` 함수는 Exp 4 커밋에 포함됨. Exp 3를 재현하려면
`select_features` 대신 `pd.concat([select_features, add_diff_features], axis=1)` 사용.

**변경 사항**: Exp 2와 모든 게 동일. feature만 변경.
- **기존(Exp 0~2)**: 센서 52개의 현재 값만 입력 → 모델이 "지금 이 순간 값들이 이상한가?"만 판단
- **Exp 3**: 현재 값 52개 + 직전 시점 대비 변화량 52개 = 104개 입력 → "지금 이상한가?" + "지금 빠르게 변하고 있는가?"를 동시에 판단

**왜 diff를 추가했나**: 화학공정 고장은 순간적으로 폭발하는 게 아니라 서서히 시작된다. 고장 초반 몇 분은 각 시점의 센서 값을 따로 떼어보면 정상 범위 안에 있어서 모델이 못 잡는다. 그러나 값이 "지금 얼마나 빠르게 올라가고 있는가(변화량)"를 함께 보면 조기에 감지할 수 있다는 가설이다. Exp 0~2에서 threshold를 아무리 정밀하게 조정해도 F1이 0.58에서 막힌 이유가 "정보 자체의 부재"라고 판단해 Tier 2(feature 추가)로 전환했다.

**왜 rolling(평균/표준편차)이 아니라 diff를 먼저 선택했나**: rolling은 "몇 개의 이전 시점을 볼 것인가(window 크기)"라는 새 하이퍼파라미터가 생긴다. 그 값을 10으로 잡든 50으로 잡든 근거가 없어서 "feature 효과"와 "window 크기 효과"가 뒤섞인다. diff는 하이퍼파라미터가 전혀 없고 "직전 대비 변화량"이라는 의미가 명확하다 — 효과 여부를 깔끔하게 검증할 수 있는 가장 단순한 방법이라 먼저 택했다.

**diff 설계 시 지킨 두 가지 규칙**:
1. `simulationRun`별로 묶어서 변화량을 계산한다. 실험 A의 마지막 시점과 실험 B의 첫 시점을 빼면 물리적으로 의미 없는 큰 값이 나오기 때문이다.
2. 각 실험의 첫 시점은 "직전 값"이 없어서 변화량이 정의되지 않는다(NaN). 이 행을 제거하면 제출 파일이 710,400행이 안 되어 채점이 깨진다 — 0으로 채운다(첫 시점은 변화 없음으로 간주).

**로컬 진단 (제출 전)**:

| | Exp 2 (기준) | Exp 3 |
|---|---|---|
| contamination | 0.23 | 0.23 (동일) |
| 입력 feature 수 | 52개 | 104개 |
| 예측 positive 수 | 256,696 | 249,321 |
| 예측 positive 비율 | 36.13% | 35.10% |
| 추정 실제 비율 | 32.20% | 32.20% |
| 비율 오차 | 4.1%p | **2.9%p** |

같은 threshold인데 모델이 "이상"이라고 판단한 행이 7,375개 줄었다. diff feature를 추가함으로써 모델이 추가 정보(변화량)를 활용해 일부 애매한 행을 "변화량은 정상이네" 하고 정상으로 돌려보낸 것으로 해석된다. 예측 비율이 실제 추정치에 2%p 더 가까워졌다 — 좋은 신호이나 F1 향상 여부는 제출 결과로만 확인 가능하다.

**점수 (public)**: F1 0.5611, Accuracy 0.7041 — Exp 2(0.5812)보다 낮고 Exp 0(0.5607)과 거의 동일

**Confusion matrix 역산** (PP=249,321):
FP+FN = 710,400 × (1-0.7041) ≈ 210,198 → TP ≈ 134,379

| | 예측 정상 | 예측 이상 |
|---|---|---|
| 실제 정상 | TN ≈ 365,823 | FP ≈ 114,942 |
| 실제 이상 | FN ≈ 95,256 | TP ≈ 134,379 |

Recall ≈ 0.5852, Precision ≈ 0.5390 (추정 실제 anomaly 비율 ≈ 0.3232, Exp 0~2와 동일)

**Exp 2 vs Exp 3 비교**:

| | Exp 2 | Exp 3 | 변화 |
|---|---|---|---|
| Recall | 0.6155 | 0.5852 | ↓ 악화 |
| Precision | 0.5506 | 0.5390 | ↓ 악화 |
| F1 | 0.5812 | 0.5611 | ↓ 악화 |

**분석 (음수 결과 — 이것 자체가 중요한 정보)**:

정밀도와 재현율이 둘 다 내려갔다. 하나가 오르고 하나가 내리는 tradeoff가 아니라 양쪽 모두 악화됐다는 건, diff feature 추가가 모델의 판단 능력 자체를 떨어뜨렸다는 뜻이다.

원인 후보 세 가지:
1. **차원 증가가 IsolationForest에 독이 됐다.** IF는 랜덤하게 feature를 선택해 분기하는데, 52개 → 104개로 늘어나면 유용한 feature가 선택될 확률이 절반으로 줄어든다. diff feature들이 "노이즈 차원"처럼 작용해 판단을 흐렸을 가능성이 높다.
2. **단 한 시점의 변화량은 정상 상태에서도 너무 시끄럽다(noisy).** 센서 값은 정상 운전 중에도 매 시점 미세하게 흔들리므로, diff의 대부분이 고장 드리프트가 아닌 배경 노이즈다.
3. **이 데이터의 고장 유형이 점진적 드리프트보다 step-change(순간 폭발적 변화) 위주일 수 있다.** Step-change 고장은 이미 절댓값에서 잘 보이므로 diff가 추가 정보를 거의 못 준다.

**결론**: `diff feature + IsolationForest` 조합은 효과 없음을 확인. "시간 정보가 부족하다"는 원래 가설은 맞을 수 있지만, IF라는 모델이 그 정보를 활용하는 구조가 아닐 가능성이 높다.

**다음 실험**: Exp 4 — max_samples=512, 원본 feature 복원

### 왜 IF를 버리고 OC-SVM으로 바꿨나

이것이 이번 실험의 핵심 결정이므로 근거를 명확히 남긴다.

**IF 클래스에서 시도한 것 전부 소진됨 (Exp 0~4)**:
- Tier 1 (threshold 튜닝): contamination `auto → 0.32 → 0.23`, F1 0.5607 → 0.5812 (+0.0205). 봉우리 확인 후 소진.
- Tier 2 feature: diff 52개 추가(104개), F1 0.5812 → 0.5611 (−0.0201). IF + diff 조합 부적합 확인.
- Tier 2 hyperparameter: max_samples 256→512, F1 0.5812 → 0.5869 (+0.0057). 마지막 의미 있는 개선.

**IF의 구조적 한계 (방법론적 근거)**:

IsolationForest는 데이터를 **수직 또는 수평으로만** 자른다. 예를 들어 "온도 > 85"처럼 하나의 축 기준으로만 경계를 긋는 것이다. 그런데 TEP의 52개 센서는 화학 반응식으로 묶여 있어, "온도가 오를 때 압력도 같이 오른다"처럼 **여러 변수가 동시에 특정 방향으로 움직인다.** 이런 상관관계를 가진 데이터에서 이상(anomaly)은 "두 변수의 관계가 틀어지는 것"으로 나타나는데, IF의 축 정렬 분기는 이것을 포착하지 못한다.

**외부 벤치마크 확인 (Hartung 2023, arXiv:2303.05904)**:

TEP 데이터에 27개 방법을 비교한 연구에서 IF는 **하위권**에 위치한다. 재구성 오차 기반 딥러닝(오토인코더, LSTM-AE)이 상위권을 차지한다. 우리의 F1 ~0.59 천장은 IF의 TEP 구조적 한계와 일치한다. 즉, 더 실험해도 IF로는 이 이상 기대하기 어렵다는 외부 근거가 있다.

**SGDOneClassSVM을 선택한 이유**:

OC-SVM은 **초평면(hyperplane)으로 경계를 긋는다** — 수평/수직이 아니라 임의의 각도. 여러 센서 값의 조합 방향으로 경계를 그을 수 있어, 센서 간 상관관계가 있는 TEP 데이터에 이론적으로 더 적합하다. 또한 `model.py`에 이미 구현되어 있어 코드 변경이 최소화된다. 딥러닝(오토인코더)보다 구현이 단순하고 실패 원인 분석이 쉬워 다음 단계의 비교 기준점으로 적합하다.

---

**변경 사항**: Exp 4와 비교해 두 가지만 변경.
- **모델**: `IsolationForest` → `SGDOneClassSVM(nu=0.23)` — nu는 IF의 contamination과 유사한 역할(임계값 조절), 첫 시도는 Exp 2~4 최적값(0.23)으로 시작
- **전처리**: `StandardScaler` 추가 — OC-SVM은 거리 기반이라 센서 단위(압력/온도/유량 등)에 따라 숫자 크기가 다르면 판단이 왜곡됨. **scaler는 반드시 train 데이터에서만 fit하고 test에는 transform만 적용한다**(data leakage 방지).

**분석**:

1. **threshold 문제 (예상됨)**: nu=0.23이 predicted rate 18.97%로 너무 보수적이다. contamination=0.23이 IF에서 36.13%를 줬던 것과 달리, nu와 predicted rate는 1:1 관계가 아니다. 재현율 0.30은 threshold 문제가 직접 원인.

2. **score 품질 문제 (더 심각한 발견)**: 예측 비율이 비슷한 조건(Exp 0 14.81% vs Exp 5 18.97%)에서 precision을 비교하면 0.8894 vs 0.5108. 즉, OC-SVM이 "이상"이라고 확신한 것들의 절반 가까이가 실제로는 정상이다. IF는 같은 비율에서 89%가 맞았다. 이건 threshold 위치 문제가 아니라 **anomaly score 자체의 분리력 문제**다.

3. **가능한 원인**: StandardScaler만으로는 TEP 센서 간 강한 비선형 상관관계를 정규화하기에 부족할 수 있다. 또는 SGDOneClassSVM의 선형 커널(기본값)이 이 데이터의 정상/이상 경계를 직선으로 충분히 표현하지 못할 가능성이 있다.

**결론**: 이 결과 하나로 OC-SVM 전체를 포기하기엔 nu 미조정 상태이므로 불공정하다. 그러나 score 품질이 이미 IF보다 낮다는 신호가 있어 큰 기대는 어렵다.

## Exp 8 — Run-level 집계 (IF Exp 4 모델 + 추론 방식 교체)

**날짜**: 2026-07-01

**재현 커밋**: (커밋 후 해시 추가 예정)
`cd src && python run_experiment.py` (EXP_NAME="exp8", RUN_CONTAMINATION=0.322)

**변경 사항**: 모델은 Exp 4와 완전히 동일(IF, contamination=0.23, max_samples=512). 추론 방식만 교체.

- **기존(Exp 0~4)**: 각 row를 독립적으로 판정 → `predict()` → binary label
- **Exp 8**: `decision_function()`으로 각 row의 연속 이상 점수를 뽑고, `simulationRun`별로 평균을 내어 run 전체를 한 번에 정상/이상 판정

**왜 이 방식인가 (t-SNE 인사이트 직접 적용)**:

t-SNE 시각화에서 이상 run의 row들이 호(arc) 모양 궤적을 이루며 정상 군집과 일관되게 분리됐다. 이는 이상 run 안의 개별 row가 하나하나 볼 때는 경계에 있더라도, 같은 run 안에서 **이상 점수가 일관되게 높다**는 의미다. 따라서 run 전체의 평균 점수로 판정하면 개별 row 판정보다 훨씬 안정적으로 이상 run을 구별할 수 있다는 가설이다.

또한 baseline EDA에서 확인된 대로 `simulationRun` 단위로 레이블이 통째로 0 또는 1이기 때문에, run 단위 판정이 문제 구조에 정확히 부합한다.

**로컬 진단 (제출 전)**:
- 예측 positive 비율: **32.16%** (추정 실제 32.20%와 0.04%p 차이 — 역대 가장 정확한 보정)
